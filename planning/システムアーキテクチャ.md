### 2.2 主要テーブル DDL

#### ハザード洪水テーブル

```sql
CREATE TABLE hazard_flood (
    id            BIGSERIAL PRIMARY KEY,
    geom          GEOMETRY(MultiPolygon, 4326) NOT NULL,
    depth_rank    SMALLINT NOT NULL,  -- 0: 0m, 1: 0-0.5m, 2: 0.5-3m, 3: 3-5m, 4: 5-10m, 5: 10m+
    depth_range   TEXT NOT NULL,
    return_period SMALLINT,           -- 再現期間（年）
    river_name    TEXT,
    source_id     INTEGER REFERENCES data_source(id),
    prefecture    TEXT NOT NULL,
    city          TEXT,
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    updated_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_hazard_flood_geom ON hazard_flood USING GIST (geom);
CREATE INDEX idx_hazard_flood_prefecture ON hazard_flood (prefecture);
```

## 3. API 処理フロー

### 3.1 メインフロー（`POST /v1/land/inspect`）

```
Client
  │
  │ POST /v1/land/inspect { address, lat, lng }
  ▼
[ALB] ── ヘルスチェック通過 ──▶ [FastAPI]
                                    │
                                    ├─ (1) API Key 認証
                                    │   └─ Redis: key_hash 照合
                                    │   └─ 失敗 → 401/403
                                    │
                                    ├─ (2) レート制限チェック
                                    │   └─ Redis: Sliding Window Counter
                                    │   └─ 超過 → 429
                                    │
                                    ├─ (3) リクエストバリデーション
                                    │   └─ Pydantic モデル
                                    │   └─ 不正 → 400
                                    │
                                    ├─ (4) キャッシュチェック
                                    │   └─ Redis: address_hash → cached_response
                                    │   └─ ヒット → (8) へスキップ
                                    │
                                    ├─ (5) ジオコーディング
                                    │   └─ address → (lat, lng)
                                    │   └─ normalize-japanese-addresses
                                    │   └─ 失敗 → lat/lng フォールバック
                                    │
                                    ├─ (6) 空間クエリ（並列実行）
                                    │   ├─ PostGIS: ST_Contains(flood, point)
                                    │   ├─ PostGIS: ST_Contains(landslide, point)
                                    │   ├─ PostGIS: ST_Contains(tsunami, point)
                                    │   ├─ PostGIS: ST_Contains(liquefaction, point)
                                    │   └─ PostGIS: ST_Contains(zoning, point)
                                    │
                                    ├─ (7) レスポンス構築
                                    │   └─ スコア計算
                                    │   └─ メタデータ付与
                                    │
                                    ├─ (8) キャッシュ保存
                                    │   └─ Redis: TTL 24h
                                    │
                                    └─ (9) 利用量記録
                                        └─ usage_log INSERT（非同期）
                                        └─ Redis: 月間カウンタ INCREMENT
```

### 3.2 ジオコーディングフロー

```
入力住所
  │
  ├─ Step 1: normalize-japanese-addresses で正規化
  │   └─ 「渋谷2-24-12」→「渋谷二丁目24番12号」
  │
  ├─ Step 2: address_master テーブルで完全一致検索
  │   └─ ヒット → (lat, lng) 取得、confidence = 1.0
  │
  ├─ Step 3: 部分一致（町丁目レベル）
  │   └─ ヒット → 町丁目の代表点を使用、confidence = 0.8
  │
  ├─ Step 4: 外部ジオコーダーにフォールバック
  │   └─ 国土地理院 API or OSM Nominatim
  │   └─ confidence = 0.6〜0.9
  │
  └─ Step 5: 全て失敗
      └─ lat/lng が指定されていればそれを使用
      └─ 両方なし → 400 エラー
```

### 3.3 空間クエリの並列実行

```python
# 概念コード
async def spatial_query(lat: float, lng: float, options: InspectOptions):
    point = f"SRID=4326;POINT({lng} {lat})"
    
    tasks = []
    if options.include_hazard:
        tasks.extend([
            query_flood(point),
            query_landslide(point),
            query_tsunami(point),
            query_liquefaction(point),
        ])
    if options.include_zoning:
        tasks.append(query_zoning(point))
    
    results = await asyncio.gather(*tasks)
    return build_response(results)
```

## 4. キャッシュ戦略

### 4.1 キャッシュレイヤー

| レイヤー | 対象 | TTL | 無効化 |
|---|---|---|---|
| **Front Door** | 静的アセット | 7日 | デプロイ時 |
| **Redis L1** | API レスポンス（住所ハッシュ） | 24時間 | データ更新時 |
| **Redis L2** | ジオコーディング結果 | 30日 | 住所マスタ更新時 |
| **Redis L3** | レート制限カウンタ | 月初リセット | — |

### 4.2 キャッシュキー設計

```
# API レスポンスキャッシュ
chiscan:inspect:{sha256(normalized_address + options_hash)}

# ジオコーディングキャッシュ
chiscan:geocode:{sha256(normalized_address)}

# レート制限
chiscan:rate:{api_key_prefix}:{YYYYMM}  # 月間カウンタ
chiscan:rate:{api_key_prefix}:{epoch_second}  # 秒間カウンタ
```

### 4.3 キャッシュヒット率の目標

- 同一住所の重複クエリ: **不動産物件は複数人が検索するため、ヒット率 40〜60% を想定**
- ジオコーディング: **住所マスタの90%以上を事前キャッシュ**

### 5.1 水平スケーリング

| コンポーネント | スケーリング方式 | トリガー |

| Phase 2 | 〜500万 | ACA 1vCPU×4〜8 | D8s_v3 + Read Replica | Standard C1 (1GB) |
| Phase 3 | 〜2,000万 | ACA 2vCPU×8〜16 | D16s_v3 + 2 Read Replicas | Premium P1 (クラスター) |
