### 1.2 サービス別構成

| サービス | Azure サービス | 構成 | 冗長性 |
|---|---|---|---|
| **ロードバランサー** | Azure Application Gateway v2 | WAF v2 統合 | ゾーン冗長 |
| **コンテナ実行** | Azure Container Apps | API コンテナ × 2（最小）→ オートスケール | ゾーン冗長 |
| **PostgreSQL** | Azure Database for PostgreSQL - Flexible Server | General Purpose D4s_v3 (4 vCPU, 16GB) | Zone-redundant HA |
| **Redis** | Azure Cache for Redis | Basic C0 → Standard C1 | 単一ノード（MVP）→ レプリカ（Phase 2） |
| **オブジェクトストレージ** | Azure Blob Storage | GIS データ保管、静的アセット、バックアップ | LRS → ZRS（99.9999999999% 耐久性） |
| **CDN** | Azure Front Door | 静的アセット + API キャッシュ + WAF | グローバルエッジ |
| **ETL** | Azure Data Factory (Managed Airflow) | Airflow 環境 | マネージド |
| **コンテナレジストリ** | Azure Container Registry (ACR) | Basic → Standard | リージョン内冗長 |

## 2. CI/CD パイプライン

### 2.1 GitHub Actions ワークフロー

```yaml
# .github/workflows/ci.yml（概念）
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  api-test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgis/postgis:16-3.4
      redis:
        image: redis:7
    steps:
      - uses: actions/checkout@v4
      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - name: Install dependencies
        run: pip install -e "api/.[dev]"
      - name: Lint (ruff)
        run: ruff check api/
      - name: Type check (mypy)
        run: mypy api/app/
      - name: Test
        run: pytest api/tests/ -v --cov=app --cov-report=xml
      - name: Upload coverage
        uses: codecov/codecov-action@v4

  web-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Install
        run: cd web && npm ci
      - name: Lint
        run: cd web && npm run lint
      - name: Type check
        run: cd web && npm run type-check
      - name: Test
        run: cd web && npm test
```

## 3. セキュリティ設計

### 3.1 OWASP Top 10 対策

| 脅威 | 対策 |
|---|---|
| **A01: アクセス制御の不備** | API Key 認証、プラン別権限制御、管理者と利用者の分離 |
| **A02: 暗号化の失敗** | HTTPS 必須、API Key のハッシュ保存（bcrypt）、DB 暗号化（Azure TDE） |
| **A03: インジェクション** | Pydantic バリデーション、SQLAlchemy ORM（パラメータバインド）、住所入力のサニタイズ |
| **A04: 安全でない設計** | レート制限、入力値バリデーション、最小権限の原則 |
| **A05: セキュリティ設定のミス** | Terraform による IaC、NSG の最小構成、デフォルト拒否 |
| **A06: 脆弱なコンポーネント** | Dependabot による自動更新、定期的な脆弱性スキャン |
| **A07: 認証の失敗** | API Key のハッシュ保存、セッション管理（JWT + 有効期限） |
| **A08: データの整合性の失敗** | データパイプラインの品質チェック、更新の原子性（テーブルスワップ） |
| **A09: ログとモニタリングの不足** | Azure Monitor Logs (Log Analytics)、アクセスログ、異常検知アラート |
| **A10: SSRF** | 内部ネットワークへのリクエストを禁止、外部 URL のホワイトリスト |

### 3.2 ネットワークセキュリティ

```
Internet
  │
  ▼
Azure Front Door (WAF Policy 連携)
  │  DDoS 保護 (Azure DDoS Protection Basic / Standard)
  ▼
Application Gateway v2 (Subnet) ※Container Apps 組み込み Ingress でも可
  │  NSG: 443 only from Front Door
  ▼
Azure Container Apps (Private Subnet)
  │  NSG: 8000 from App Gateway only
  │  VNet 統合で Private Endpoint 経由でデータ層にアクセス
  ▼
PostgreSQL / Redis (Private Endpoint)
  │  NSG: 5432/6379 from Container Apps Subnet only
  │  インターネットアクセス不可
  │  Private Link で VNet 内のみ到達可能
```

### 3.3 データセキュリティ

| 対象 | 保護方式 |
|---|---|
| API Key（保存時） | bcrypt ハッシュ |
| API Key（転送時） | HTTPS (TLS 1.3) |
| DB データ（保存時） | Azure Database for PostgreSQL の TDE（AES-256、Azure Key Vault 統合） |
| DB 接続 | SSL 必須 |
| Blob Storage データ | Azure Storage Service Encryption（SSE、AES-256） |
| ユーザーパスワード | bcrypt + salt |
| ログ | Azure Monitor Logs（Log Analytics ワークスペース暗号化） |
| シークレット | Azure Key Vault に一元管理（API Key, DB接続文字列, 外部API鍵） |

### 3.4 API セキュリティ

| 対策 | 実装 |
|---|---|
| **認証** | X-API-Key ヘッダー |
| **レート制限** | Redis Sliding Window Counter |
| **入力バリデーション** | Pydantic v2 strict mode |
| **CORS** | 許可オリジンのホワイトリスト |
| **レスポンスヘッダー** | X-Content-Type-Options, Strict-Transport-Security |
| **リクエストサイズ制限** | 1MB（バッチ: 10MB） |
| **IP ブロック** | Front Door WAF ルールで異常トラフィックを自動ブロック |

## 5. バックアップ・DR

### 5.1 バックアップ

| 対象 | 方式 | 頻度 | 保持期間 |
|---|---|---|---|
| PostgreSQL | 自動バックアップ（Azure マネージド） | 連続（差分） | 7日間（MVP）→ 35日間（Phase 2） |
| PostgreSQL | Point-in-Time Recovery (PITR) | 連続（5分間隔） | 最大35日間 |
| Blob Storage (GIS データ) | Soft Delete + バージョニング | 自動 | 永続 |
| Blob Storage (ETL ログ) | ライフサイクル管理ポリシー | — | 90日 → Cool → Archive |
| Redis | RDB スナップショット（Standard 以上） | 日次 | 設定に準ずる |

### 5.2 DR（災害復旧）

| 目標 | MVP | Phase 2 以降 |
|---|---|---|
| **RPO**（復旧時点目標） | 24時間 | 5分（PITR） |
| **RTO**（復旧時間目標） | 4時間 | 1時間 |
| **DR 方式** | 単一リージョン + バックアップ復元 | Zone-redundant HA + Read Replica |

> Azure Database for PostgreSQL - Flexible Server は Zone-redundant HA を標準でサポート。  
> Phase 2 以降は Japan West へのジオレプリケーションも検討可能。

## 7. Azure Container Apps 選定理由

| 要件 | Container Apps | AKS | App Service |
|---|---|---|---|
| **サーバーレス（インフラ管理不要）** | ✅ | ✗（クラスタ管理要） | △ |
| **オートスケール（0→N）** | ✅（KEDA ベース） | ✅（要設定） | △ |
| **コスト（MVP規模）** | ✅（従量課金、低コスト） | ✗（最低コスト高め） | △ |
| **VNet 統合** | ✅ | ✅ | ✅ |
| **カスタムドメイン + SSL** | ✅ | ✅ | ✅ |
| **リビジョンベースデプロイ** | ✅（Blue-Green容易） | ✅ | △ |
| **PostGIS 対応（Docker 自由度）** | ✅ | ✅ | ✗ |
| **将来の AKS 移行** | 容易（Kubernetes ベース） | — | 困難 |

> **判断**: MVP〜Phase 2 は Container Apps で運用コストを最小化。  
> Phase 3 以降でトラフィック増大時に AKS への移行パスが明確。
