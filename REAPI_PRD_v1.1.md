## 1. エグゼクティブサマリー

ChiScan は、住所・緯度経度を入力するだけで、ハザードリスク・都市計画規制・用途地域を一括返却する日本初の土地調査統合プラットフォームである。

**提供形態は2層構造で設計する。**

| レイヤー | 形態 | ターゲット | マネタイズ |
|---|---|---|---|
| **Consumer Layer** | GUI Webアプリ（無料） | 一般ユーザー・不動産検討者・個人事業主 | 広告収益 |
| **Developer Layer** | REST API（有料） | 企業・開発者・AIエージェント | SaaS サブスク・従量課金 |

GUIはAPIのエントリーポイントとして機能し、「まず無料で使ってみる → APIの価値を体感 → 有料API契約」というファネルを形成する。

> **ミッション: 日本の土地情報を、誰でも・どこからでも・1秒で取得できる社会インフラを作る。**

## 2. 解決する問題

### 2.1 現状の課題

日本の公的土地データは以下の理由でアクセスが極めて困難である。

- **データソースの分散**: ハザード情報は国交省、用途地域は各市区町村、都市計画は都道府県がそれぞれ管理
- **フォーマットの不統一**: 自治体ごとに GIS 仕様・CSV 構造・凡例定義が異なる
- **更新頻度の不透明性**: 最終更新日が明記されていないデータが多数存在
- **API 未提供**: 大半はダウンロード式 CSV または PDF 配布のみ
- **AI 非対応**: 構造化されていないため LLM・AI エージェントからの自動利用が不可能

### 2.2 ユーザーの現在のワークフロー（As-Is）

```
① 国土交通省ハザードマップポータルにアクセス
② 住所を手入力して地図上で目視確認
③ 自治体の都市計画 GIS ページに移動
④ 用途地域・建ぺい率・容積率を手転記
⑤ 社内の調査票に記入
→ 合計 30〜90 分 / 件
```

### 2.3 解決後のワークフロー（To-Be）

```
【一般ユーザー】
① ChiScan Web で住所を検索
② 結果をその場で確認（0秒）

【開発者・企業】
① 住所を ChiScan API に POST
② JSON レスポンスを受け取り自社サービスに組み込む（< 1秒）

【AI エージェント】
① MCP 経由で ChiScan を呼び出し
② 構造化データをそのまま判断に利用
```

## 3. ターゲット市場

### 3.1 市場セグメント

| セグメント | 想定企業・ユーザー数 | 主な利用形態 | 優先度 |
|---|---|---|---|

| 不動産仲介・売買 | 約 12 万社 | API | 最優先 |
| 建築設計事務所 | 約 2 万社 | API | 高 |

| 損害保険（引受審査） | 約 30 社 | API（大量バッチ） | 超高単価 |

| AI エージェント開発者 | 急増中 | MCP / API | 将来の柱 |

### 3.2 ターゲットペルソナ

#### ペルソナ A：マイホーム検討中の会社員（GUI ユーザー）
- **課題**: 候補物件の土地リスクを調べたいが専門知識がなく行政サイトが複雑
- **求めること**: 住所を入れたら一瞬でわかる、無料で使える
- **ChiScan との接点**: Google 検索「〇〇市 ハザードマップ 調べ方」から流入
- **マネタイズ**: 住宅ローン・引越し・保険などの広告表示

#### ペルソナ B：不動産テックスタートアップ CTO
- **課題**: 物件詳細ページにハザード情報を表示したいが自前でデータ収集するコストが高い
- **求めること**: 月額定額・高速・信頼性のある API
- **成功基準**: 実装工数 1 日以内、レスポンス 500ms 以下

#### ペルソナ C：損保会社データエンジニア
- **課題**: 引受審査の自動化に洪水・土砂リスクを取り込みたいが公的データの整形コストが大きい
- **求めること**: SLA 保証・監査ログ・大量バッチ処理
- **成功基準**: 月 100 万件処理、データの法的根拠が明確

#### ペルソナ D：AI エージェント開発者
- **課題**: LLM エージェントに「この土地は安全か」を判断させたいが構造化データがない
- **求めること**: 機械可読な JSON・スキーマ安定・MCP サーバー対応
- **成功基準**: エージェントから直接呼べる・フィールド名が変わらない

### 4.2 GUI Web アプリ（Consumer Layer）

#### 4.2.1 主要機能

| 機能 | 内容 | MVP |
|---|---|---|
| 住所検索 | 住所入力 → ハザード・用途地域を即時表示 | ✅ |
| 地図表示 | ハザードゾーンのオーバーレイ表示 | ✅ |
| リスクサマリー | わかりやすい5段階スコア表示 | ✅ |
| PDF レポート出力 | 調査結果の PDF ダウンロード | Phase 2 |
| 比較機能 | 複数住所の並列比較 | Phase 2 |
| 履歴保存 | 過去の検索履歴（ログイン時） | Phase 2 |

#### 4.2.2 広告マネタイズ戦略

GUI の無料ユーザーは「土地・住宅購入を検討中」という高い購買意向を持つ。このインテントシグナルを活用した広告配置を設計する。

| 広告枠 | 内容 | 想定 CPM / CPC |
|---|---|---|
| 住宅ローン比較 | 検索結果ページの上部バナー | 高単価（住宅金融）|
| 火災・地震保険 | ハザードスコア表示後の自然文脈広告 | 高単価（損保） |
| 引越し・リフォーム | レポート下部 | 中単価 |
| 不動産仲介 | 周辺物件情報と連動 | 中単価 |

> **広告設計の原則**: ユーザー体験を損なわないよう、検索・結果表示フローには広告を挟まない。広告は結果表示後の「次のアクション」文脈にのみ配置する。

#### 4.2.3 API への誘導ファネル

```
一般ユーザーが無料 GUI を利用
    ↓
「API でシステム連携できます」のバナー表示
    ↓
開発者向けランディングページ
    ↓
Starter プラン（無料・1,000件/月）で試す
    ↓
有料 API 契約へのコンバージョン
```

## 5. データ戦略

### 5.1 データソースと収集方法

| データ種別 | ソース | 収集方法 | 更新頻度 |
|---|---|---|---|
| 洪水ハザード | 国土交通省ハザードマップポータル | 公開 GIS データ取得 | 年 1〜2 回 |
| 土砂災害警戒区域 | 国土交通省・都道府県 | シェープファイル取得 | 年 1〜2 回 |
| 津波浸水想定 | 内閣府・都道府県 | 公開データ取得 | 数年単位 |
| 液状化リスク | 各自治体 GIS | スクレイピング＋手動整備 | 数年単位 |
| 用途地域 | 国土数値情報ダウンロード | 公開 GIS データ取得 | 年 1 回 |
| 建ぺい率・容積率 | 用途地域に付随 / 自治体条例 | GIS + 条例テキスト | 年 1 回 |
| 防火・準防火地域 | 国土数値情報 | 公開 GIS データ取得 | 年 1 回 |
| 登記情報（将来） | 法務局・登記情報提供サービス | API 連携交渉 / スクレイピング | リアルタイム〜週次 |

### 5.2 データパイプライン

```
収集 (Crawler / Downloader)
  ↓
正規化・標準化 (ETL: 座標系統一 → JGD2011、フォーマット統一)
  ↓
品質チェック (カバレッジ確認 / 異常値検知)
  ↓
PostGIS (空間DB: ポリゴン管理)
  ↓
API レイヤー (住所 → 座標変換 → 空間クエリ → JSON 返却)
  ↓
GUI レイヤー (地図 + リスクサマリー表示)
```

### 5.3 住所正規化

- 大字・字の省略パターン対応
- 枝番・号・方書き対応
- IMI コンポーネント（デジタル庁 OSS）をベースに独自拡張
- マッチ失敗時は lat/lng フォールバックを許容

## 6. 技術スタック

| レイヤー | 技術選定 | 理由 |
|---|---|---|
| API サーバー | FastAPI (Python) | 地理データ処理ライブラリが豊富 |
| GUI フロントエンド | Next.js + Maplibre GL | SSR による SEO 最適化・地図描画性能 |
| 空間データベース | PostgreSQL + PostGIS | ポリゴン空間クエリの業界標準 |
| 座標変換 | pyproj / PROJ | JGD2011 ↔ WGS84 変換 |
| ジオコーディング | IMI コンポーネント + 自社拡張 | 国内住所に特化 |
| キャッシュ | Redis | 同一住所の重複クエリ削減 |
| ETL パイプライン | Apache Airflow | 定期データ更新の管理 |
| 広告配信 | Google Ad Manager | 高単価広告枠（住宅・金融）へのアクセス |
| インフラ | AWS (Tokyo リージョン) | データ主権・低遅延 |
| 認証 | API Key + JWT | 開発者向け標準 |

## 8. フェーズ別ロードマップ

### Phase 1：MVP（〜3ヶ月）

### Phase 2：全国展開（〜6ヶ月）
- 全国 47 都道府県カバー完了
- GUI の全国対応・SEO 強化
- バッチ処理 API
- Business / Enterprise プラン開始
- 金融・保険向けセールス開始

**成功指標**: 有料契約 50 社 / GUI 月間 UU 10 万 / ARR 2,500 万円

### Phase 3：AI 対応（〜12ヶ月）
- MCP サーバー提供
- AI エージェント向けスキーマ最適化
- GUI に AI チャット機能追加（「この土地は安全ですか？」に自然言語で回答）
- 変更履歴 API

**成功指標**: 有料契約 150 社 / GUI 月間 UU 30 万 / ARR 8,000 万円

### Phase 4：登記情報拡張（〜18ヶ月以降）

> **長期ビジョン**: ChiScan が土地調査のデファクトになった後、登記情報・所有者情報へと展開し「土地に関するあらゆる情報を返す API」へと進化させる。

| 拡張項目 | 内容 | ハードル |
|---|---|---|
| 登記情報 API | 所有者・抵当権・面積・地目 | 法務局との連携交渉、法的整備 |
| 所有者変遷履歴 | 売買・相続の履歴 | データ取得コスト・プライバシー |
| 抵当権・差押情報 | 金融機関向け審査支援 | 法的取り扱い要確認 |
| 隣接地情報 | 隣接する土地の一括調査 | 空間クエリ拡張で技術的には可能 |

> **登記情報の参入戦略**: 直接スクレイピングではなく、①法務省・デジタル庁との公式連携交渉、②登記情報提供サービス（法務局公式）との API 接続、③弁護士・司法書士事務所との B2B パートナーシップ、の順で進める。規制と信頼性が最大の課題であり、フェーズ 1〜3 での実績・信頼構築が前提条件となる。

**成功指標**: ARR 1.5 億円 / 登記 API β 版リリース / 業界標準スキーマ OSS 公開

## 9. 競合分析

| 競合 | 提供内容 | 弱点 | ChiScan の優位性 |
|---|---|---|---|
| 国土交通省（直接） | 生データのダウンロード | API なし・整形不可・AI 非対応 | 使える状態にする |
| Google Maps API | 地図・ルート・住所変換 | ハザード・用途地域は非提供 | 国内特化データ |
| SUUMO / LIFULL | 物件情報 API | 自社プラットフォーム囲い込み | 汎用 B2B API |
| 不動産 SaaS 各社 | 業務ソフト内蔵データ | API として外部提供せず | 外部連携対応 |
| 海外（Hazard Hub 等） | 米国向けハザード API | 日本データ非対応 | 日本語・日本法規準拠 |

> **結論**: 「日本の土地データをAPIとして外部提供する専業プレイヤー」は2026年4月時点で存在しない。

## 10. リスクと対策

| リスク | 深刻度 | 発生確率 | 対策 |
|---|---|---|---|
| デジタル庁が無料 API を整備する | 高 | 中 | 整備に 5〜10 年かかる。その間に市場を取る。整備後も「正規化・SLA・GUI」の価値は残る |

| 登記情報の法的ハードル | 高 | 高 | Phase 4 以降に先送り。Phase 1〜3 の実績を持って交渉に臨む |

## 12. Go-to-Market 戦略

### 12.1 GUI による自然流入（SEO ドリブン）
